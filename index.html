C‑IDS User Guide

What C‑IDS is and how the pieces fit together.

C‑IDS is a modular, cross‑platform Intrusion Detection System core. The detection logic lives once, in C++, and every platform talks to it through a stable binary interface instead of keeping its own copy of the logic.

There is no server and no telemetry. Every packet is inspected on the device it runs on, and nothing leaves that device.

HOW IT'S STRUCTURED

  1. 1. Core (C++20) — parses traffic and matches it against detection rules behind a stable extern "C" boundary.
  2. 2. .NET bindings — a type‑safe P/Invoke layer wraps the core for Windows desktop apps, using SafeHandle for lifetime management.
  3. 3. Swift bindings — a Swift Package connects iOS apps to the core through a C module map and ships as an .xcframework.
  4. 4. Full architecture notes, the ABI contract, and the security policy live in the docs/ folder of the repository.

MODULE STATUS

3/3 ONLINE
CORE

C++ Engine

A platform-independent detection engine, built on a zero-trust memory policy, RAII discipline, and a stable C ABI boundary.

  • › C++20 · header/impl separation
  • › no raw owning pointers
  • › ASan / UBSan verified
.NET

Desktop Bindings

A type-safe P/Invoke layer for Windows. Wraps the native engine with SafeHandle discipline and never leaks a raw pointer.

  • › LibraryImport source-gen
  • › SafeHandle lifetime
  • › C# / .NET 8
SWIFT

Mobile Bindings

A Swift Package for iOS. Connects to the core through a C module map, so application code never sees an unsafe pointer.

  • › Swift Package Manager
  • › .xcframework distribution
  • › iOS 15+

DESIGN PRINCIPLES

The same principles apply on every platform.

ABI stability first

Only the extern "C" surface is contractual; the C++ internals are free to change.

Zero‑trust memory

No raw owning pointers — every boundary is validated.

Local‑first

No server, no telemetry — analysis stays on the device.

Plugins are guests

The core has no knowledge of any plugin's runtime.
